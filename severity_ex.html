Severity 판단 Worked Examples

KWCAG 2.2 기반 · AT 사용자 관점 · 멘탈 모델 v2.0 · 5 Cases

각 케이스에서 판단자의 사고 흐름을 단계별로 따라가세요. ⚠️ 판단 포인트에 특히 주목하세요 — 초심자가 틀리기 쉬운 지점을 짚어줍니다. 판단 경로를 먼저 보고 싶다면 결론 영역의 판단 경로 칩을 확인하세요.
Critical
KWCAG 2.2 · 1.1.1 대체 텍스트

인포그래픽 이미지에 빈 대체 텍스트(alt="") 적용

의미 있는 정보를 담은 이미지가 장식 이미지로 잘못 처리된 경우
시나리오
뉴스 앱 기사 본문 중간에 경제 지표 통계가 담긴 인포그래픽 이미지가 삽입되어 있다. 해당 이미지 마크업은 <img src="infographic.png" alt="">로 처리됨. 본문 텍스트에는 "아래 그래프를 참고하세요"라는 문장이 있고, 인포그래픽의 실제 수치 정보는 이미지 안에만 존재한다.
Step 0 판단 전 프레이밍 — 타겟 AT 유저와 달성 목표를 먼저 설정한다
타겟 AT 유저
전맹 시각장애인
스크린리더(JAWS, VoiceOver) 사용자
달성 목표
인포그래픽에 담긴 경제 지표 수치 파악
기사 내용을 온전히 이해하기 위한 핵심 정보
Step 1 AT 사용자가 해당 콘텐츠·기능을 이용할 수 있는가?
판단 쓸 수 없다
alt=""는 스크린리더에게 "이 이미지는 장식용이므로 건너뛰세요"라는 신호다. JAWS와 VoiceOver 모두 해당 이미지를 완전히 무시하며, 키보드 탐색 시 이미지 위치조차 지나쳐버린다. AT 사용자는 인포그래픽이 존재한다는 사실 자체를 알 수 없고, 당연히 그 안의 수치 정보에 접근할 방법이 없다.
많은 초심자가 "화면에 이미지가 보이니까 이용 가능하다"고 판단하기 쉽다. 그러나 평가 기준은 항상 타겟 AT 유저의 관점이다. 스크린리더 사용자에게 alt="" 이미지는 존재하지 않는 것과 같다.
Step 2 무엇인지 알 수 있는가? AT가 콘텐츠·기능의 목적·내용을 인식 가능한가?
판단 알 수 없음
이미지 자체가 인식되지 않으므로 그 안의 정보 존재를 알 수 없다. 본문에 "아래 그래프를 참고하세요"라는 텍스트가 있어 그래프의 존재를 추론할 수는 있다. 그러나 이는 달성 목표인 "경제 지표 수치 파악"을 실현하지 않는다. 수치 정보 자체에 접근할 방법이 전혀 없다.
"아래 그래프를 참고하세요"라는 주변 텍스트는 이미지의 존재를 암시하지만, 달성 목표를 이루게 하지는 않는다. 이는 캐러셀 인디케이터(●○○)가 페이지 존재를 알려주지만 내용 파악은 불가능한 것과 같은 논리다. 판단 기준은 항상 "달성 목표를 이룰 수 있는가"이지, "존재를 알 수 있는가"가 아니다.
결론 Critical
판단 경로
이용 불가 → 알 수 없음 → Critical
근거 한 줄
스크린리더가 이미지 존재 자체를 인식하지 못해 경제 지표 수치를 알 방법이 전혀 없음.
Major A
KWCAG 2.2 · 2.1.1 키보드 사용 보장

서식 툴바 버튼이 마우스 클릭으로만 작동, 키보드 접근 불가

기능의 목적은 인식 가능하지만 AT로 실행 자체가 차단된 경우
시나리오
이메일 서비스의 텍스트 편집기에서 서식 툴바(굵게·기울임·밑줄 버튼)가 div + JavaScript 클릭 핸들러로 구현되어 있다. tabindex가 없어 키보드 Tab으로 포커스 이동이 불가능하며, 마우스 클릭으로만 서식을 적용할 수 있다. 버튼들은 화면에 "B / I / U" 레이블로 시각적으로 명확히 표시되어 있다.
Step 0 판단 전 프레이밍
타겟 AT 유저
운동장애 · 키보드 전용 사용자
손떨림·근력 저하로 마우스 조작 불가. 시각은 정상.
달성 목표
이메일 본문에 Bold 서식 적용
강조할 문장을 굵게 처리하는 편집 작업
Step 1 AT 사용자가 해당 콘텐츠·기능을 이용할 수 있는가?
판단 쓸 수 없다
Tab 키로 편집기 영역을 탐색해도 서식 버튼에 포커스가 이동하지 않는다. 키보드로 Bold 버튼을 활성화하는 방법 자체가 없다. (본 시나리오에서는 에디터가 브라우저 기본 단축키 Ctrl+B도 오버라이드하여 작동하지 않는다고 가정한다.)
키보드 사용자라고 해서 반드시 시각장애인이 아니다. 운동장애 사용자는 화면을 직접 보면서 키보드만 사용할 수 있다. "Bold 버튼이 화면에 보인다"는 사실은 목적 인식에는 해당하지만, 기능 실행의 가능 여부와는 별개다.
Step 2 무엇인지 알 수 있는가? AT가 콘텐츠·기능의 목적을 인식 가능한가?
판단 알 수 있음
화면에 "B"(Bold) 버튼이 시각적으로 명확하게 표시되어 있다. 사용자는 이 버튼이 굵게 서식을 적용한다는 목적을 인식하고 있다. 문제는 그 목적을 알 수 없는 것이 아니라, AT(키보드)로 실행할 수 없는 것이다.
여기가 Critical과 Major A의 분기점이다. Critical은 "뭔지도 모르고 쓸 수도 없는" 상태, Major A는 "뭔지는 알지만 쓸 수 없는" 상태다. 목적 인식 가능 여부가 두 등급을 가른다.
결론 Major A
판단 경로
이용 불가 → 알 수 있음 → Major A
근거 한 줄
서식 버튼의 목적은 시각적으로 파악 가능하나, 키보드(AT)로 실행 자체가 불가능한 상호작용 차단 상태.
Major B
KWCAG 2.2 · 3.3.1 오류 식별

폼 입력 오류를 CSS 빨간 테두리로만 표시, 텍스트 메시지 없음

이용 불완전 + 대안 조건 미충족 + 상황 파악 불가로 Major B에 도달하는 경우
시나리오
온라인 서비스 회원가입 폼에서 이메일 형식 오류·비밀번호 조건 미충족 등 입력 오류 발생 시, 오류가 있는 필드의 CSS 테두리 색상만 빨간색으로 변경된다. 오류를 설명하는 텍스트 메시지는 DOM에 존재하지 않으며 aria-describedby, role="alert" 등 ARIA 처리도 없다.
Step 0 판단 전 프레이밍
타겟 AT 유저
전맹 시각장애인
스크린리더(JAWS, VoiceOver) 사용자
달성 목표
회원가입 폼 오류 수정 후 제출 완료
어떤 필드에 어떤 오류가 있는지 파악하고 수정하는 전체 흐름
Step 1 AT 사용자가 해당 콘텐츠·기능을 이용할 수 있는가?
판단 접근 가능, 이용 불완전
스크린리더로 폼 필드 탐색·텍스트 입력·제출 버튼 클릭까지 모두 가능하다. 그러나 제출 후 오류가 발생했을 때, 어떤 필드에 어떤 문제가 있는지를 스크린리더가 읽을 수 없다. 기능 자체(폼 작성)는 이용 가능하지만 오류 식별·수정 단계에서 이용이 불완전해진다.
"폼 접근 자체가 불가능한가?"와 "폼의 특정 흐름에서 이용이 막히는가?"를 구분해야 한다. 이 케이스는 폼 접근은 가능하지만 오류 수정 플로우에서 AT 사용자에게만 장벽이 생기는 경우다.
Step 2 신뢰 가능한 대안이 있는가? 대안 인정 2조건을 확인한다
판단 대안 없음
대안 인정 기준 체크
✗
① AT 자체 기능으로 달성 목표를 완전히 실현할 수 있는 경우 해당 없음 — 스크린리더는 CSS 색상 변화를 읽지 않음. 오류 정보가 DOM 텍스트로 없으므로 어떤 AT 기능도 접근 불가.
✗
② 설계된 다른 UI 경로로 동일한 달성 목표에 도달할 수 있는 경우 해당 없음 — 오류 피드백을 전달하는 별도 UI 경로가 설계되지 않았음. 대안 UI 자체가 없음.
사용자가 모든 필드를 하나씩 재입력해보는 방법은 "사용자 수준 워크어라운드"다. 이는 대안 인정 기준 ①, ② 어디에도 해당하지 않아 대안으로 인정하지 않는다. 대안 판단의 핵심은 "AT 또는 설계된 경로로 달성 목표를 완전히 실현 가능한가"다.
Step 3 회복 가능하고 상황 파악이 가능한가?
판단 아니오
어떤 필드가 오류인지, 오류가 몇 개인지조차 스크린리더로 알 수 없다. 제출에 실패했다는 사실(예: 페이지 새로고침)은 감지할 수 있을 수 있으나, 어떤 필드를 어떻게 수정해야 하는지 상황 파악이 불가능하다. 목표(회원가입 완료) 달성 방향을 전혀 알 수 없는 상태다.
"회복 가능하다"는 것은 "다시 시도할 수 있다"는 것이 아니다. 방향 없이 반복 시도하는 것은 회복이 아니다. "어떻게 해야 하는지 알 수 있는가"가 회복 가능성의 핵심이다.
결론 Major B
판단 경로
이용 불완전 → 대안 없음 → 상황 파악 불가 → Major B
근거 한 줄
오류 정보가 DOM에 없어 스크린리더로 어떤 필드를 수정해야 하는지 상황 파악 자체가 불가능한 이용 불완전 상태.
Minor
KWCAG 2.2 · 2.4.3 초점 이동

모달 오픈 시 포커스가 모달 내부로 이동하지 않음

이용 불완전이지만 AT 자체 기능으로 달성 목표를 완전히 실현 가능한 경우
시나리오
서비스 가입 페이지에서 "이용약관 보기" 버튼 클릭 시 약관 내용이 담긴 모달이 열린다. 그러나 포커스(focus)가 모달 내부로 이동하지 않고 "이용약관 보기" 버튼에 그대로 남아있다. 모달 마크업은 role="dialog"와 aria-modal="true"가 올바르게 적용되어 있고, 약관 텍스트와 동의 체크박스가 DOM에 정상적으로 존재한다.
Step 0 판단 전 프레이밍
타겟 AT 유저
스크린리더+키보드 사용자
시각장애, 또는 운동장애+시각장애 복합 사용자
달성 목표
이용약관 내용 확인 및 동의 체크박스 선택
가입 완료를 위한 필수 단계 — 약관 전문 읽기 + 체크 선택
Step 1 AT 사용자가 해당 콘텐츠·기능을 이용할 수 있는가?
판단 접근 가능, 이용 불완전
모달이 DOM에 존재하고 role="dialog"가 적용되어 있으므로 스크린리더가 모달을 탐색할 수 있다. 그러나 모달 오픈 시 포커스가 자동으로 이동하지 않아 사용자가 즉시 약관 내용을 읽을 수 없고, 모달이 열렸다는 사실을 즉각적으로 알기 어렵다. 불완전하지만 이용이 완전히 막힌 것은 아니다.
"모달이 열린지 즉시 알지 못한다" ≠ "모달을 이용할 수 없다". 스크린리더의 가상 커서(Browse Mode)는 페이지 전체를 탐색할 수 있으므로, 즉각적 피드백이 없다고 해서 바로 "이용 불가"로 판단하지 않는다.
Step 2 신뢰 가능한 대안이 있는가? 대안 인정 2조건을 확인한다
판단 대안 있음
대안 인정 기준 체크
✓
① AT 자체 기능으로 달성 목표를 완전히 실현할 수 있는 경우 해당 — VoiceOver·JAWS의 가상 커서(Browse Mode)로 페이지 내 모든 요소 탐색 가능. role="dialog" 랜드마크로 모달에 직접 이동 가능(F6, VO+F8 등). 약관 텍스트 전체 읽기 및 동의 체크박스 선택까지 AT 자체 기능으로 완전히 달성 가능.
이것이 Minor와 Major B의 분기점이다. "가상 커서로 탐색"은 스크린리더가 기본 제공하는 핵심 탐색 방식으로 대안 인정 기준 ①에 해당한다. 추가 설치나 우회 방법이 아닌 AT의 본래 기능으로 달성 목표를 완전히 실현할 수 있다면 Minor다.
결론 Minor
판단 경로
이용 불완전 → 대안 있음 (①) → Minor
근거 한 줄
초점 자동 이동 미구현으로 탐색 불편이 생기나, AT 가상 커서로 모달 내 약관 확인과 동의 선택을 완전히 실현할 수 있음.
Trivial
KWCAG 2.2 오류 유형 목록 외

캐러셀 컴포넌트에 role="carousel" 적용 — ARIA 명세에 없는 역할값

문법적으로 잘못된 마크업이지만 AT 사용자 경험에 실질적 영향이 없는 경우
시나리오
메인 페이지 이미지 캐러셀 배너의 최상위 컨테이너에 role="carousel"이 적용되어 있다. carousel은 WAI-ARIA 1.2 명세에 정의되어 있지 않은 역할값이다. 배너 내 각 슬라이드는 <li> 요소로 마크업되어 있고, 슬라이드 안의 텍스트·이미지 대체 텍스트·링크는 모두 DOM에 정상적으로 존재한다.
이 오류는 KWCAG 2.2의 84개 오류 유형 목록에 포함되지 않습니다. QA 과정에서 발견된 마크업 문법 오류이지만, ARIA 명세 위반이 직접적인 접근성 장벽을 만들지 않아 KWCAG 2.2 점검 항목으로 분류되지 않은 케이스입니다. 점검 중 목록 외 이슈가 발견될 경우, 동일한 Severity 판단 모델을 적용하여 분류할 수 있습니다.
Step 0 판단 전 프레이밍
타겟 AT 유저
스크린리더 사용자
VoiceOver(macOS/iOS), JAWS, NVDA 사용자
달성 목표
캐러셀 배너 슬라이드 콘텐츠 탐색 및 내용 파악
각 슬라이드의 텍스트·이미지 정보 인식
Step 1 AT 사용자가 해당 콘텐츠·기능을 이용할 수 있는가?
판단 실질 영향 없음
WAI-ARIA 1.2 명세에 carousel이라는 역할값은 정의되어 있지 않다. 스크린리더는 알 수 없는 role 값을 무시하고, 해당 요소의 기본 HTML 시맨틱(<ul> → 목록)으로 폴백(fallback)한다. 결과적으로 JAWS·VoiceOver 모두 슬라이드 내부의 텍스트·대체 텍스트·링크를 정상적으로 읽는다. 사용자가 콘텐츠를 탐색하고 내용을 파악하는 달성 목표에 아무런 장벽이 생기지 않는다.
"마크업 오류가 있다 = 접근성 문제다"로 등치하는 실수가 흔하다. Trivial 판단의 핵심은 "AT 사용자의 실제 경험이 달라지는가"이지, 명세 준수 여부 자체가 아니다. 코드 품질 관점의 오류와 사용자 경험 관점의 접근성 오류는 구분해야 한다. 단, 동일한 마크업 오류라도 다른 컨텍스트에서는 실질 영향이 생길 수 있으므로 항상 AT 동작을 기준으로 판단한다.
결론 Trivial
판단 경로
실질 영향 없음 → Trivial
근거 한 줄
role="carousel"은 ARIA 명세에 없어 AT에 무시되며, 슬라이드 콘텐츠 탐색·내용 파악에 실질적 장벽이 없음.
